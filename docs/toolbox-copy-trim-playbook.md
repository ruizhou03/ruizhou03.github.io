# 工具文案精简 Playbook —— 多余提示性文字审计

> 内部流程文档（`docs/` 已在 `_config.yml` 里 exclude，不会上线）。
> 用于逐个扫描百宝箱（`toolbox/`）里的工具，删掉多余的提示性文字，或在“该靠 UI 表达”的地方优化 UI 来精简文字。
> 缘起：宠物中心（`/toolbox/pet/`）首次实践，详见本仓库 commit `b1f6d15`。

## 目的（为什么做这件事）

界面上常驻的提示性文字过多时，会稀释信息、显得啰嗦、增加阅读负担。**很多提示用户其实能通过 UI 设计或上下文自行理解**——这类文字就是多余的，应当删除；如果删了会影响理解，那往往说明 UI 本身没把意思表达清楚，应该去优化 UI，让信息靠界面（图标、布局、占位符、分组、可见的交互手柄）表达，而不是靠一段说明文字。

一句话原则：**能让用户凭 UI 或上下文自行意会的，就不该常驻在界面上。**

## 适用范围

`toolbox/` 下任意一个工具，**一次只扫一个**。每个工具走一遍下面的三步法。

---

## 三步法（判断框架）

### 第一步：盘点工具里所有“用户可见的文字”

先把用户实际能看到的文字全部列出来。来源有三类，三类都要扫：

1. **静态结构 HTML** —— `toolbox/<工具>/index.html`、`_includes/toolbox/<工具>/*.html`
2. **弹窗 / 模态框** —— 通常也在上面的 include 里（modals 之类）
3. **JS 动态插入的文字** —— `assets/js/<工具>.js`，重点找：
   - 空状态（“还没有…”、“暂无…”）
   - toast / 横幅 / 内联提示
   - 校验 alert（只在出错时出现）
   - 动态副标题、动态按钮文字
   - `placeholder` 占位符
   - 动态设的 `title` 工具提示
4. （少见）CSS `::before/::after` 的 `content` —— `assets/css/<工具>.css`

常用 grep 起手式（按需调整工具名）：
```bash
# 静态文字
grep -nE "placeholder=|title=|class=\"hint" _includes/toolbox/<工具>/*.html toolbox/<工具>/index.html
# JS 里的中文说明句 / 空状态 / 口语化连接词
grep -noE "['\"\`][^'\"\`]*(比如|可以|就能|即可|记得|别忘|先|两次|拖动|换算|不确定|默认|大约|还没|暂无|提示|说明)[^'\"\`]*['\"\`]" assets/js/<工具>.js
```

**区分“提示性文字”和“功能性文字”**——只审前者：

| 类型 | 是否在审计范围 | 例子 |
|---|---|---|
| 提示性文字（解释 / 引导 / 说明） | ✅ 审 | hint 说明句、步骤提示、空状态鼓励语、操作说明、领域知识注解 |
| 功能性文字（UI 固有标签） | ❌ 一般不动 | 按钮文字、字段 label、section 标题、数值 / 单位、选项名 |

（功能性文字若本身啰嗦也可顺手精简，但它们不是这次的主要目标。）

### 第二步：对每一处提示性文字，自问

> **“如果删了这个提示性文字，会不会影响用户体验？用户会不会不理解这个按钮、设计或者步骤是在做什么？”**

### 第三步：按答案分流

- **答“不会影响，用户能完全意会”** → **大胆删除。**
- **答“会影响”** → 不能直接删。两条路：
  1. **保留**（这是真有用的信息）；或
  2. **先优化 UI**，让信息改由界面表达，之后再删掉这段文字。

## 判断启发（从宠物中心实战提炼）

### 倾向「删」的信号

- **行动入口 / 答案就在旁边**，提示却在重复它。
  例：空状态“还没有宠物——先建一只吧”，正下方就是「+ 添加第一只」按钮 → 删尾巴，留“还没有宠物”。
- **被其它 UI 元素重复表达**的步骤 / 进度说明。
  例：向导“第 2 步 / 共 3 步 · 选记录方式”，上方已有进度圆点 + 每步大标题 + 字段 label → 整条删。
- **重述显而易见操作**的话（“点这个按钮来保存”之类）。
- **空状态里的鼓励尾巴**（“…就能看到曲线啦”），当 CTA 已经明摆着时。

### 倾向「留」的信号（删了**会**影响理解）

- **用户不查就不知道的领域知识**：典型数值范围、专业术语、“去哪找这个数据”。
  例：“看粮袋上的热量，一般 3.5–4.2；不确定就按默认。”
- **非显然行为 / 状态的当场解释**：在那一刻说明“刚发生了什么、为什么是这个结果”。
  例：记录起点后 toast“（下次再记就能算出吃了多少）”——解释这次为什么不算“吃了”。
- **只在出错 / 特定状态出现的反馈**：校验提示、审批结果、权限说明。这类本就不常驻，留着。
- **区分两个相似选项的小例子**：帮用户一眼选对。
  例：“按个/根/份（猫条·冻干·整罐）”vs“按克数（用秤称的主粮）”。

### 「先优化 UI 再删」的常见手法

当一段说明确实有用、但又不想常驻占地方时，把它换成自解释的 UI：

- **可折叠的 ⓘ 小图标**：常驻只留一个 info 图标，点开才展开说明。适合大段领域说明。
- **占位符承载引导**：把“去哪找 + 示例”塞进 `placeholder`，省掉独立 hint 段。
  例：`placeholder="包装上找「热量/能量」，比如 80"`。
- **更自解释的图标 / 布局 / 分组**：用可见的交互手柄、图标、视觉分组替代“告诉你能这么操作”的文字。
- **示例值 / 单位后缀**：用 `≈ 结果` 实时预览或单位 suffix 替代解释性句子。

## 执行流程（每个工具一轮）

1. **定位文件**：`toolbox/<工具>/`、`_includes/toolbox/<工具>/`、`assets/js/<工具>.js`、`assets/css/<工具>.css`。
2. **盘点**：按第一步把所有用户可见提示文字列出来。
3. **逐条判断**：按第二、三步分类。建议产出一张三列表：**已删 / 该留（附原因）/ 需用户拍板**。
4. **动手**：
   - **明显安全**的（用户百分百能意会）→ 直接删。
   - **边界**的（删了影响很小、靠个人偏好）→ **不要自作主张，列给用户用具体选项拍板**（“删了影响很小但要不要留一句友好引导”这种）。
   - **删了会影响理解**的 → 保留，并在汇报里写明“若要删，需先怎样优化 UI”。
5. **本地验证**：改完用常驻的 `jekyll serve --livereload`（localhost:4000）核对。
6. **提交**：守仓库《Git 提交边界》（见根目录 `CLAUDE.md`）——只 `git add` 自己这轮改的具体文件；工作区有别人 WIP / 本地与 origin 分叉时，用隔离 `git worktree` 从 `origin/main` 落一条干净提交。

## 验证清单（改完逐项确认）

```bash
html=$(curl -s http://localhost:4000/toolbox/<工具>/)
# 1) 该删的确实没了
echo "$html" | grep -qF "<被删的句子>" && echo "✗ 还在" || echo "✓ 已删"
# 2) 该留的核心文字还在
echo "$html" | grep -qF "<核心标签/说明>" && echo "✓ 保留"
# 3) 页面能正常构建（curl 200、无 Liquid 报错）
```

- [ ] 删的都消失了
- [ ] 留的核心标签 / 说明都在
- [ ] 页面正常渲染、功能未被误删（尤其别删到带 `id=` 的元素本身，只删文字）
- [ ] 边界项都经用户确认，没有自作主张删
