# 安全与恢复基线

> 当前权威的安全状态摘要。它不含密钥值，不替代服务商控制台。最后核对：2026-07-20。

## 已确认的数据边界

| 数据 | 存储 | 敏感度 |
|---|---|---|
| 公开文章、工具源码、公开资料 | GitHub / Pages | 公开 |
| 账号、邮箱、Google `sub`、积分、收藏、阅读记录、游戏存档 | Upstash Redis | 个人信息 / 用户资产 |
| 密码 scrypt 哈希 | Upstash Redis | 高敏感认证数据 |
| 评论、回复、评论邮箱、页面阅读计数 | Waline PostgreSQL | 个人信息 / 用户内容 |
| 上传图片、音频 | Cloudflare R2 | 公开内容与用户上传混合 |
| JWT、数据库、R2、OAuth、AI 等密钥 | fly secrets / GitHub Actions secrets / 本机 Keychain | 信任根，绝不入 Git |

评论库不是 Upstash。任何只备份 Redis 的方案都不算完整备份。

## 当前控制

- GitHub Pages 强制 HTTPS；主仓库公开，因此保密内容必须同时受 `.gitignore` 与 `_config.yml exclude` 约束。
- 后端源码工作副本、内部文档、邮件摘要均不进入 Pages 构建；旧付费系统已于 2026-07-20 退役。
- JWT 使用 HS256；2026-07-20 已删除公开的默认密钥，`AUTH_JWT_SECRET` 缺失时后端拒绝启动。线上部署仍须先核实 fly secret。
- 新密码注册及 Google-only 账号反向设置密码均已关闭；旧密码账号只有验证当前密码后才能改成 12–128 位新密码，邮箱在关联 Google 前标记为未验证。
- Google token 严格验签并按稳定 `sub` 绑定；同名邮箱不再自动并入未验证密码账号。
- 账号已有部分数据导出和注销能力，但尚未覆盖 comments Postgres、R2 与全部业务键，不得宣称为全量删除。
- CORS `*` 是当前 Bearer-token 模型下的刻意约束；若未来改 cookie 鉴权，必须同步收紧。
- Redis、PostgreSQL 与 R2 的每周备份和恢复工具位于 `scripts/backup/`；2026-07-20 首次自动备份、上传回读和完整临时环境恢复均成功，可恢复性已标绿，下一次演练最迟 2026-10-20。
- 公开隐私说明位于 `/privacy/`，全站页脚和个人中心均有入口。
- 全站流量主口径为 Cloudflare Web Analytics；GA4 仅两个独立首页，不作为全站总量。

## 同名邮箱冲突恢复

1. 若 Google 登录返回 `account_link_required`，且用户记得旧密码：先用邮箱密码登录，再在同一会话中点 Google 验证。后端只在密码凭证和同名 Google ID token 同时成立时绑定 `googleSub`。
2. 若用户不掌握旧密码：禁止自动合并。管理员先备份旧账号哈希及关联键，通过实时 Google 登录确认邮箱归属，再将旧账号隔离并为已验证 Google `sub` 建立新账号。
3. 不把旧账号的私密数据自动复制给新账号；只有用户能同时证明旧密码与 Google 控制权时才允许原位关联。人工恢复必须记录备份、决策与执行人。

## 备份的安全要求

1. 业务 R2 bucket 与备份 bucket 必须不同；备份 bucket 保持私有。
2. Redis 与 PostgreSQL 备份上传前必须加密；加密口令在 GitHub secret 之外另存一份。
3. 自动工作流必须做格式校验、上传后下载回读和 SHA-256 核对。
4. 镜像 R2 时不使用 `--delete`，避免源端误删同步破坏备份。
5. 恢复工具默认 dry-run / 拒绝覆盖；破坏性恢复要求显式确认短语。
6. “备份成功”不等于“能恢复”，季度演练不可省略。
7. 演练用 R2 token 只授权备份桶和临时桶，演练后删除临时桶、清除本机临时凭据，并由账号主人在 Cloudflare 控制台撤销 token。

## 文档保密说明

根目录可能存在未跟踪的 `SECURITY_GOVERNANCE.md` 草稿，它不属于当前权威文档，也不能在未审阅时提交。`_config.yml` 已预防性排除同名文件，避免未来误进入 Pages；即便被排除，主仓库仍是 public，提交前仍要确认内容适合公开。
