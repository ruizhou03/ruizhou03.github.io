# 领地卡大厅设计模板（「跳棋模式」）

> 本文是一次长对话里挖掘并确立为准绳的 UI 设计模板的交接文档。
> 目标：把这套「开局/选座」设计推广到其它棋类 / 联机游戏。
> 写于 2026-06，准绳实现 = 跳棋(tiaoqi) + 飞行棋(feixingqi)。

## 0. 一句话

把「开局 / 选座」从**居中弹窗**，改成**在棋盘上每位玩家的专属领地里放一张配置卡**。
卡 = 领地的底框，点卡循环 `本机真人 / 电脑(难度) / 邀请联机`。

## 1. 适用前提（什么游戏能用）

硬性前提：**每位玩家在盘面上拥有一块可视的专属区域（领地 / 基地 / 半盘）**。

| 能用 | 不能用 |
|---|---|
| ✅ 跳棋（6 个三角家）— 模板原型 | ❌ 五子棋 / 四子棋 / 黑白棋（共享同一网格，无归属区） |
| ✅ 飞行棋（4 个角落方形基地） | ❌ 斗地主 / 掼蛋（牌桌四周的「座位」，非盘面领地） |
| ✅ 中国象棋 / 国际象棋（各占半盘，楚河汉界 / 中线分界） | ❌ 任何「双方在同一片区域落子」的游戏 |

**判据**：如果你能在盘面上指着说「这块是红方的」，就能用；否则不能。
五子棋类即使硬要用也找不到领地，先放弃。

## 2. 准绳参考实现

- **跳棋** `assets/js/games/tiaoqi.js` + `toolbox/tiaoqi/index.html`（三角领地，最完整）
- **飞行棋** `toolbox/feixingqi/index.html`（方形基地，跳棋的原型，`fxq-` 与 `tq-` 类名一一对应）

这两个是「领地卡 + 座位/颜色绑定 + 各自视角」的活样板，新游戏直接照抄它们，别从零想。

## 3. 卡片的形态准则

- **卡 = 领地底框**：卡的形状要贴合领地形状并**铺满它**——别搞成「大领地框 + 中间一个小矩形文字卡」两层套着。
  - 三角领地 → 圆角三角；方形 / 半盘 → 圆角矩形。
- **三角领地框做法（跳棋）**：用 **Minkowski（三角形 ⊕ 半径 `pad` 的圆盘）** 生成圆角三角路径——三条边各向外平移 `pad` 清开棋子，三个尖角变成「以角点格子为圆心、半径 `pad` 的圆弧」（圆角顺着棋盘格点）。
  - `pad` 取值原则：**盖住本领地的棋子，又不压到领地外最近一排格子**。跳棋实测 `pad=21`（棋子半径 18）。
- **卡内容（统一）**：图标 + 队名(红方/黑方/…) + 状态(本机真人/电脑/邀请联机) + [电脑时]难度档一行 + [邀请时]房号/复制链接/状态。
- **卡循环顺序**：`本机真人 → 电脑 → 邀请联机 →（回到本机真人）`。

## 4. 已知坑（务必避开，都是这次踩出来的）

1. **难度档竖排**：盘面卡用 `position:absolute; left:NN%` 居中定位时，`width:auto` 会按「stage 宽 − left」收缩适配，右侧领地分到的宽很小 → 按钮文字被挤成竖排。
   **解法**：`.seat { width: max-content; max-width: …% }` + 难度按钮 `white-space: nowrap; flex-wrap: nowrap`。
2. **联机着色脱节**：盘面着色 / active 判定，**联机时必须读 `net.seats`（实际座位），不能读本地 `setupConfig`**。否则大厅里盘面颜色与座位脱节（已占座位没着色、空位反而着色）。飞行棋、跳棋都犯过这个，统一改 `activeCornerSet()` 联机分支读 seatMap。
3. **底框留缝**：见 §3——卡要铺满领地，文字浮在卡上，别留中间矩形缝。
4. **二维码 / 提示文字多余**：邀请只给「复制链接」，**不要二维码**；盘面别堆操作提示文字（靠卡本身和禁用态表达即可）。
5. **开始按钮位置**：2 人棋（象棋 / 国象）把「开始游戏」放在**中线（楚河汉界）那条空带**上；进入邀请态也别把它移走 / 别塞进卡里。

## 5. 邀请联机流程（tiaoqi 式，准绳）

点「邀请联机」那一刻（**在点击手势里同步完成**）：

1. 随机生成 4 位房号；
2. **立刻把邀请链接写进剪贴板**（必须在手势内，否则浏览器拦截剪贴板写入）；
3. 乐观地把房号显示在卡里（不等后端）；
4. **后台**用这个房号建真房间；撞号(`code_taken`)就换一个号重试 + 重新复制。

被邀请者：进 `?room=XXXX` 链接 → 按**掼蛋逻辑直接分配到房主邀请的那一侧座位**，用领地卡呈现自己的位置，等房主开局。
（视角绑定在**开局时**才落定，大厅里换位不立即改视角。）

## 6. 联机引擎（让「邀请」真能对弈）

后端：`https://zircon-urge.fly.dev/api/board`（通用「房间 + 事件日志」服务，与 gameId 无关）。

**已直连 curl 验证过的后端事实**：

- 动作：`create / join / start / move / state / reorder / leave / kick / event`。
- `deviceId` 必须是 UUID 格式（用 `GamesShell.Identity.getDeviceId()`，乱填报 `did_invalid`）。
- 座位返回 `seat: 'first' | 'second'`；**后端强制轮替**：seat `first` 先走，同一人连走报 `not_your_turn`。
- `reorder` **只在 `state==='lobby'` 可用**（开局后报 `not_lobby`）。
- 走子日志 `room.moves = [{playerId, move:{from,to} | event:{…}, ts}]`。

**引擎五件套**（照 chess `toolbox/chess/index.html` 或 tiaoqi 抄）：

1. **走子同步**：本地落子 `POST move {from,to}`；轮询 `state`，把 `room.moves` 里**新出现的对手条目**回放（`applyMove(m,{fromRemote:true})`，远端来的不再广播）。
2. **颜色绑定**：座位 `first = 先手色`（象棋红 / 国象白）、`second = 后手色`。房主按「邀请的是哪一侧」**在大厅内 `reorder`**，让先手色坐 `first`（因为后端规定 first 先走）。这样房主能自由选任意一侧邀请。
3. **各自视角**：`isFlipped()` 联机时按「我方颜色」翻盘，己方棋子永远在下方。
4. **回合限制**：非我方回合 / 非我方子，点击直接 return。
5. **抢动回滚**：`POST move` 返回 `not_your_turn` 时，把刚才本地那步 `undo` 回去再重渲染。

**计数对齐坑**：`movesApplied` 进局时设成 `room.moves.length`（把开局前的事件条目算进去），之后自己每步**广播成功**后 `++`，否则会把自己的走子当对手的再 apply 一遍。

**开始按钮门控**：需要 `reorder` 的那一侧（房主邀请红），要等 reorder 落定后才允许点开始（否则座位顺序没改好就开局，颜色会错）。

## 7. 开局翻盘动画（待做）

开局时把棋盘转到「我方在下方」的朝向，带一个旋转动画。
**注意**：象棋棋子是汉字，纯 180° CSS 旋转会让字倒过来——需要旋转动画结束后，再以「正立的翻转渲染」落定（旋转只是过场，最终态是正立 + 坐标翻转）。

## 9. 推广到一个新游戏的步骤

1. 确认它满足 §1（盘面上有专属领地）。不满足就别硬上。
2. 抄 tiaoqi / feixingqi 的卡渲染 + 循环 + 形态（§3）。
3. **若它联机已完整**（如 chess）：把领地卡接到现成的颜色绑定 / 视角即可，工作量小。
4. **若它联机是空壳**（如 xiangqi 曾经）：先按 §6 建引擎，再接 UI。
5. 全程避开 §4 的坑；邀请走 §5；开始按钮按 §4.5。

## 10. 运维须知（别再被这些坑到）

- **部署 = push 到 `main`**（GitHub Pages 只构建 `main`，绑定 ruizhou03.com）。
- **「看不到更新」≠「没部署」**：本站是 PWA，`sw.js` 对 HTML 是 network-first 但有 **4 秒兜底**——慢网 / 大页面（象棋 ≈158KB）会先吐**缓存旧页**、后台再更新缓存。
  - 验证是否真部署：`curl https://ruizhou03.com/toolbox/<game>/`（绕过 SW，看源码里有没有新标记）。
  - 用户侧拿最新：**无痕窗口**（无 SW，必新且联机可用）／点页面「新版本·刷新缓存」指示／多刷两次／DevTools 注销 SW。
- **提交边界**：本仓库多会话并发，**只 `git add` 自己这次改的具体文件**，禁止 `git add -A`；push 前看清 `git log origin/main..HEAD`，别把别人的 WIP / 提交一起推上去。
